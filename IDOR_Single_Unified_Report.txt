Finding 1: Insecure Direct Object Reference (IDOR) - Multiple Endpoints
Critical

Dependency: Application

Status: Open

Description: The KPMG Tax Calculator application contains a systemic Insecure Direct Object Reference (IDOR) vulnerability affecting multiple endpoints. Authenticated users can access arbitrary client data without proper authorization checks by manipulating numeric ID parameters. An attacker with valid session credentials can enumerate all client IDs and retrieve sensitive data belonging to other organizations. The application validates user authentication (via TC-BAC-DEV cookie) but fails to implement authorization checks that verify users have permission to access requested resources.

This is a multi-endpoint vulnerability affecting at least three critical endpoints:
1. /gettaxcalcbac/ClientTaxLogic
2. /gettaxcalcbac/ClientSelect/ViewClient
3. /gettaxcalcbac/CalcTransactions

During testing, a user with clientId=98 (Ankan Bose) was able to successfully access sensitive data for clientId=30 (Sudeep Dwivedi's organization), clientId=741, and multiple other client IDs by simply changing the numeric parameter value. The application responded with HTTP 200 and returned full page content with complete transaction records, tax configurations, and client information without any permission denial.

This vulnerability represents a complete breakdown of multi-tenant data isolation and exposes:
- Tax jurisdiction selections and configurations
- Award type and plan code settings
- Client transaction records and calculation results
- Compensation and equity award data
- Advanced variable defaults
- Exchange rate configurations
- Client-specific business intelligence

The vulnerability is systematic and fundamental to the application's authorization architecture, not isolated to individual endpoints.

Remediation: 

IMMEDIATE ACTION REQUIRED (CRITICAL - Must Fix Before Production):

ROOT CAUSE: The application lacks authorization checks on all endpoints. Developers assumed that validating user authentication (login) was sufficient, but failed to implement authorization checks that verify users have permission to access specific resources.

1. Implement Authorization Checks on All Vulnerable Endpoints

Generic Code Pattern (applies to all 3 endpoints):

BEFORE (Vulnerable):
    public ActionResult ClientTaxLogic(int id)
    {
        // WRONG: Only checks authentication, no authorization
        // Assumes logged-in user can see any client data
        var data = db.GetClientData(id);
        return View(data);
    }

AFTER (Secure):
    public ActionResult ClientTaxLogic(int id)
    {
        // Step 1: Get current authenticated user
        var currentUser = GetCurrentUser(); // Gets Ankan Bose
        
        // Step 2: Get list of clients user is authorized to access
        var authorizedClientIds = currentUser.GetAllowedClientIds(); 
        // Returns: [98] for Ankan
        
        // Step 3: Verify requested ID is in authorized list
        if (!authorizedClientIds.Contains(id))
        {
            // User does not own this client - DENY ACCESS
            return new HttpStatusCodeResult(HttpStatusCode.Forbidden);
        }
        
        // Step 4: Only retrieve data if authorization passed
        var data = db.GetClientData(id);
        return View(data);
    }

Apply this same pattern to ALL three endpoints:
- ClientTaxLogic endpoint
- ClientSelect/ViewClient endpoint
- CalcTransactions endpoint

2. Create User-Client Mapping in Database

SQL Example:
    CREATE TABLE UserClientAccess (
        UserId INT,
        ClientId INT,
        AccessLevel VARCHAR(50), -- READ, WRITE, ADMIN
        CreatedDate DATETIME,
        PRIMARY KEY (UserId, ClientId)
    );
    
    -- Example: Ankan can only access client 98
    INSERT INTO UserClientAccess VALUES (1, 98, 'READ', GETUTCDATE());
    
    -- Example: Sudeep can only access client 30
    INSERT INTO UserClientAccess VALUES (2, 30, 'READ', GETUTCDATE());

3. Implement Authorization Helper Method

    public class AuthorizationService {
        
        public bool UserCanAccessClient(int userId, int clientId)
        {
            using (var context = new AppDbContext())
            {
                var access = context.UserClientAccess
                    .Where(x => x.UserId == userId && x.ClientId == clientId)
                    .FirstOrDefault();
                
                return access != null; // True if user has access, False otherwise
            }
        }
    }

4. Add Audit Logging

Track all access attempts (successful and failed):

    public void LogClientAccess(int userId, int clientId, bool success, string ipAddress)
    {
        var auditLog = new AuditLog
        {
            UserId = userId,
            ClientId = clientId,
            AccessResult = success ? "SUCCESS" : "DENIED",
            Timestamp = DateTime.UtcNow,
            IpAddress = ipAddress,
            Endpoint = "ClientTaxLogic"
        };
        
        context.AuditLogs.Add(auditLog);
        context.SaveChanges();
        
        // Alert if unauthorized access attempted
        if (!success)
        {
            AlertSecurityTeam($"Unauthorized access attempt: User {userId} tried to access Client {clientId}");
        }
    }

5. Replace Numeric IDs with UUID References (Long-term Fix)

BEFORE (Guessable):
    /gettaxcalcbac/ClientTaxLogic?id=30

AFTER (Non-guessable):
    /gettaxcalcbac/ClientTaxLogic?clientRef=a7f2c5d1-8b3e-4f6a-9c2e-1d5b8f3a6c9e

Mapping Table:
    CREATE TABLE ClientReferences (
        ClientRefId UNIQUEIDENTIFIER, -- UUID
        ActualClientId INT,
        UserId INT,
        CreatedDate DATETIME
    );

This prevents enumeration since UUIDs are:
- Not sequential
- Not guessable
- Unique per user-client relationship

6. Implement Input Whitelist Validation

    public ActionResult ClientTaxLogic(int id)
    {
        var currentUser = GetCurrentUser();
        var validIds = context.UserClientAccess
            .Where(x => x.UserId == currentUser.Id)
            .Select(x => x.ClientId)
            .ToList();
        
        // Whitelist validation: only allow IDs in user's list
        if (!validIds.Contains(id))
        {
            throw new UnauthorizedAccessException($"Client {id} not in your authorized list");

    }

7. Add Rate Limiting

Prevent automated enumeration attacks:

    public class RateLimitingFilter : ActionFilterAttribute
    {
        public override void OnActionExecuting(ActionExecutingContext filterContext)
        {
            var userId = GetCurrentUser().Id;
            var ipAddress = filterContext.HttpContext.Request.UserHostAddress;
            
            // Allow max 100 requests per hour per user
            var recentRequests = context.RequestLog
                .Where(x => x.UserId == userId && x.Timestamp > DateTime.UtcNow.AddHours(-1))
                .Count();
            
            if (recentRequests > 100)
            {
                filterContext.Result = new HttpStatusCodeResult(HttpStatusCode.TooManyRequests);
            }
        }
    }

Root Cause Analysis:

The developers made a critical security mistake:

❌ INCORRECT ASSUMPTION:
   "If a user is logged in, they can access any client data"
   (Authentication = Authorization)

✅ CORRECT APPROACH:
   "User is logged in AND user owns the requested resource"
   (Authentication + Authorization Required)

This is a fundamental misunderstanding of multi-tenant security architecture. Every multi-tenant application MUST verify ownership of resources for each request.

Conclusion:

The KPMG Tax Calculator contains a CRITICAL, SYSTEMIC IDOR vulnerability affecting multiple endpoints. This is not an isolated bug but a fundamental architectural flaw in the authorization implementation. The application successfully authenticates users but completely lacks authorization checks to verify resource ownership.

This vulnerability allows any authenticated user to access sensitive data belonging to any other organization in the system. The ease of exploitation combined with the sensitivity of the data and the scope of affected organizations makes this a CRITICAL severity issue requiring immediate remediation.
