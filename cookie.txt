Finding 5: Cacheable SSL Pages Found
Risk Rating: Low
Dependency: Application
Status: Open
Description
The application fails to enforce proper cache control directives on SSL/HTTPS pages containing sensitive information. Pages transmitted over secure HTTPS connections are being cached by the browser, which allows sensitive data to remain accessible in the browser cache.
This vulnerability arises from the absence of or improper implementation of cache control headers (Cache-Control, Pragma, Expires) that explicitly prevent caching of sensitive content. When users access authentication pages, user profile pages, or pages containing personal/financial data over HTTPS, the browser stores these pages in its local cache.
An attacker with physical access to the user's computer, or a malicious user on a shared computer, can access the browser's cache to retrieve previously visited sensitive pages without requiring valid authentication credentials. The vulnerability is particularly concerning in environments with:

Shared workstations
Public computers (internet cafes, libraries)
Corporate environments where multiple users share the same device

Additionally, cached HTTPS content may be accessible through browser history, cached images, or forensic analysis of the storage medium.
Affected Systems & Locations
This vulnerability affects multiple critical pages across the application:

https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/login/index (Authentication page - highly sensitive)
https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/login/enter (Login processing page)
https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/Home/QuickSearch (User dashboard with personal data)
https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/ClientTaxLogic (Client configuration data)
https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/CalcTransactions (Transaction records and calculations)
https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/ClientSelect/ViewClient (Client information)

All authenticated user pages and sensitive data endpoints are affected. Testing revealed that HTTP response headers do not include proper cache control directives. Pages are cached with default browser settings, allowing indefinite retention in the browser cache unless manually cleared by the user.
Reproduction Steps
Step 1: Access the application login page

Open browser (Chrome, Firefox, Edge, Safari)
Navigate to https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/login/index
Open Developer Tools (F12 or Right-click → Inspect)
Navigate to Network tab

Step 2: Check response headers for cache control

Observe the HTTP response headers for the login page
Look for the following headers: Cache-Control, Pragma, Expires
Expected (SECURE): Cache-Control: no-store, no-cache, must-revalidate, private
Expected (SECURE): Pragma: no-cache
Observed (VULNERABLE): Headers are missing or set to allow caching
Observed (VULNERABLE): Cache-Control: public or Cache-Control: max-age=3600

Step 3: Verify page is cached

Complete login process with valid credentials
Log out or clear session
Close the browser completely
Reopen the browser
Navigate to https://qa.gmsgetb.kpmglinkgo.com/gettaxcalcbac/login/index

Step 4: Access cached page without authentication

Open browser cache/history
Using keyboard shortcut Ctrl+H (History), search for the domain
Alternatively, use browser developer tools to inspect cached resources
Observe: Previously visited authenticated pages are accessible from cache
Observe: Page content displays without requiring re-authentication
FINDING: Sensitive pages are retained in browser cache

Step 5: Verify sensitive data in cache

Navigate to authenticated pages containing user data (e.g., /Home/QuickSearch)
Check browser cache location:

Windows: C:\Users[username]\AppData\Local[BrowserName]\Cache
macOS: ~/Library/Caches/[BrowserName]
Linux: ~/.cache/[BrowserName]


Observe: HTML files contain sensitive information including transaction details, client data, tax calculations
FINDING: Sensitive user data persists in cache files

Remediation
Implement Cache-Control Headers on All Sensitive Pages:
Add the following HTTP response headers to all pages containing sensitive information:
Cache-Control: no-store, no-cache, must-revalidate, private, max-age=0
Pragma: no-cache
Expires: 0
Set-Cookie headers should include:

HttpOnly
Secure
SameSite=Strict

Implementation Guidelines:

Configure cache-control at application level (web server or application framework)
Apply to all endpoints that handle:

User authentication
Login/logout operations
User profile and account information
Transaction records
Client data
Administrative functions
Session information


Implement for both Pre-Auth (login pages) and Post-Auth (all authenticated pages)
