Finding 7: Exposed Unminified Application Source Code
================================================================================

SEVERITY: HIGH

Dependency: Application

Status: Open

Description:
During penetration testing of the gettaxcalcbac application, a critical 
vulnerability was identified in the client-side JavaScript file "appMaster.js". 
The file is served unminified and unobfuscated, exposing the complete 
application logic, internal functions, API endpoints, and form structures.

The file contains 50+ global functions with human-readable names including 
searchGETID(), switchClient(), openPage(), and clientPressed(). All form field 
names are visible (quickSearchType, quickSearchValue, quickSearchClientId, etc.), 
and API endpoint URLs are hardcoded in clear text.

An unauthenticated attacker can completely reverse-engineer the application 
by analyzing this source code, identifying all security weaknesses and 
potential attack vectors. An authenticated attacker can leverage this 
information to exploit multiple vulnerabilities including:

  • Multi-tenant data isolation bypass via switchClient() function
  • Session token theft via exposed cookie names
  • Form manipulation and parameter injection
  • Real-time search query interception and logging
  • Device fingerprinting for permanent user tracking
  • Persistent malicious code injection via localStorage

The combination of exposed source code with missing security controls 
(no CSRF tokens, no server-side authorization checks, no rate limiting) 
creates a perfect environment for complete system compromise.

Technical Details:

VULNERABLE CODE EXAMPLE 1 - Multi-Tenant Switching:

function switchClient(clientId, clientCode) {
    var elmnt = document.getElementsByClassName('js-switch-client-id');
    var elmntCode = document.getElementsByClassName('js-switch-client-code');
    if (elmnt != null) {
        elmnt[0].value = clientId;
        elmntCode[0].value = clientCode;
        elmnt[0].form.submit();
    }
}

Problem: Function is publicly accessible, takes any clientId as parameter,
no authorization check visible. Allows switching to any of 12 client accounts.

VULNERABLE CODE EXAMPLE 2 - Search Query Exposure:

function searchGETID(searchText, clientId) {
    console.log("searchGETID:" + searchText);
    document.quickSearchForm.action = rootUrl + "Home/QuickSearch";
    document.quickSearchForm.method = "post";
    document.quickSearchForm.quickSearchType.value = "GETID";
    document.quickSearchForm.quickSearchValue.value = searchText;
    document.quickSearchForm.quickSearchClientId.value = clientId;
    document.quickSearchForm.submit();
}

Problem: Function name and parameters are exposed, form field names are visible,
console logging reveals search terms. Attacker can intercept and exfiltrate.

VULNERABLE CODE EXAMPLE 3 - Unprotected Fingerprinting:

new Fingerprint2().get(function (result, components) {
    $.ajax({
        type: "POST",
        url: rootUrl + "WebService/SetFingerprinting",
        data: { fingerprinting: canvasFingerprinting },
        success: function (responseHtml) { },
        error: function (data) { }
    });
});

Problem: AJAX call is unprotected, no CSRF token, API endpoint is exposed,
fingerprinting data sent without user consent.

Remediation:

IMMEDIATE (1 Week) - CRITICAL:
1. Minify appMaster.js
   Action: Use Terser or UglifyJS to minify JavaScript
   Impact: Reduces file from 15KB to ~5KB, code becomes unreadable
   Timeline: 1-2 days

2. Remove Console Logging
   Action: Strip all console.log() statements
   Impact: Prevents search term exposure
   Timeline: 1 day

3. Remove Source Maps
   Action: Verify no .map files served in production
   Impact: Prevents source code reconstruction
   Timeline: 1 day

4. Deploy Updated Code
   Action: Deploy minified version to production
   Impact: Immediate risk reduction
   Timeline: Same day

SHORT TERM (1 Month) - HIGH:

Closing/Downgrading Justification:

This vulnerability should NOT be downgraded below HIGH severity until:

  ✓ JavaScript is minified in production
  ✓ All sensitive information is removed from client-side code
  ✓ CSRF tokens are implemented on all forms
  ✓ Server-side authorization checks are implemented
  ✓ Rate limiting is implemented
  ✓ Security headers are added
  ✓ Persistent code injection is prevented (CSP)
  ✓ Session tokens are protected with HttpOnly flag

Downgrading this vulnerability without addressing the underlying issues would
be irresponsible given the critical nature of the exposures and the ease of
exploitation demonstrated in the POCs.
